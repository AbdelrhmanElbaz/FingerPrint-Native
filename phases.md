# phases.md — خطة المراحل والاختبار

كل مرحلة **مستقلة وقابلة للاختبار بمفردها** قبل الانتقال للتي تليها. لا تنتقل لمرحلة جديدة قبل التأكد من نجاح اختبارات المرحلة الحالية.

---

## Phase 0 — الهيكل الأساسي والتغليف (Skeleton + Packaging Pipeline)

**الهدف:** إثبات أن السلسلة كاملة (كود → exe) شغّالة من أول يوم، قبل كتابة أي منطق فعلي.

- إنشاء المشروع بالهيكل الموضح في `claude.md`.
- نافذة `QMainWindow` فارغة تفتح فعليًا، بعنوان البرنامج والأيقونة.
- إعداد `PyInstaller` وإنتاج أول `exe` تجريبي.

**✅ اختبار القبول:**
- [ ] الـ exe يفتح على جهاز Windows نظيف (بدون Python مثبَّت) بدون أخطاء.
- [ ] حجم الـ exe معقول (يُسجَّل كخط أساس للمقارنة لاحقًا).
- [ ] زمن بدء التشغيل (Startup Time) يُقاس ويُسجَّل كمرجع.

---

## Phase 1 — طبقة البيانات (Database Layer)

**الهدف:** قاعدة SQLite شغّالة بكل الجداول المذكورة في `schema.md`، مع الـ Repositories الأساسية.

- إنشاء الجداول (عبر SQLAlchemy models أو SQL خام + migration بسيط).
- تنفيذ `CompanyRepository`, `EmployeeRepository`, `AttendanceFileRepository` (CRUD أساسي فقط).
- اختبار الإنشاء التلقائي لقاعدة البيانات أول تشغيل.

**✅ اختبار القبول:**
- [ ] Unit tests لكل Repository (إضافة/قراءة/تعديل/حذف شركة، ملف شهر، موظف).
- [ ] إغلاق البرنامج وإعادة فتحه → البيانات المحفوظة موجودة كما هي (لا فقدان بيانات).
- [ ] اختبار Concurrency بسيط: فتح وإغلاق البرنامج عدة مرات متتالية بدون تلف للقاعدة.

---

## Phase 2 — الاستيراد والتحليل (Parsing Engine) — **الأهم في المشروع**

**الهدف:** نقل منطق تحليل ملفات البصمة (Hikvision + الصيغة القديمة) حرفيًا، مع اختبارات Regression صارمة.

- نقل `parse_file_hikvision`, `parse_file`, `redistribute_overnight_punches`, `saturate_punches`, `assign_shift_patterns`, `classify_lone_punch` إلى `services/parsers/`.
- تخزين النتائج في قاعدة البيانات (بدل إرجاع `emp_days` كـ dict في الذاكرة فقط).

**✅ اختبار القبول (الأهم في كل الخطة):**
- [ ] جمع 5-10 ملفات إكسل حقيقية (من الاستخدام الفعلي للعميل) كـ Fixtures.
- [ ] تشغيل نفس الملفات على النسخة القديمة (Streamlit) وتسجيل النتائج (ساعات العمل، أيام الحضور، أيام الغياب لكل موظف) كـ "Golden Output".
- [ ] تشغيل نفس الملفات على النسخة الجديدة والتأكد من **تطابق تام** للأرقام مع الـ Golden Output.
- [ ] اختبار حالات حرجة: شيفت يمتد بعد منتصف الليل، بصمة واحدة فقط في اليوم، يوم بدون بصمات، ملف بأيام شهر مختلفة (28/29/30/31).

> ⚠️ لا تنتقل لمرحلة تالية قبل اجتياز هذه الاختبارات 100% — أي فرق في رقم هنا يعني خطأ فادح يؤثر على راتب حقيقي لموظف حقيقي.

## Phase 3 — واجهة شجرة الملفات والتنقل (Sidebar Navigation)

**الهدف:** التنقل بين الشركات/السنوات/الشهور، فتح/حذف/إعادة تسمية.

- `QTreeWidget` للشجرة + قائمة سياقية (Context Menu).
- شاشة الاستيراد (Import View) + "فتح مؤقت (Anonymous)".
- استعادة آخر جلسة عند فتح البرنامج.

**✅ اختبار القبول:**
- [ ] إنشاء شركة جديدة → استيراد ملف → يظهر في الشجرة فورًا.
- [ ] حذف شهر / حذف شركة كاملة → يختفي من الشجرة والقرص معًا.
- [ ] إعادة تسمية شركة → الاسم يتحدّث في كل مكان (شجرة + عنوان النافذة).
- [ ] إغلاق البرنامج وإعادة فتحه → يقترح فتح آخر ملف تم العمل عليه.
- [ ] فتح ملف بوضع Anonymous → التأكد أنه لا يُحفظ في القاعدة نهائيًا بعد إغلاق البرنامج.

## Phase 4 — لوحة التحكم والرواتب (Dashboard + Payroll Calculation)

**الهدف:** نقل `calculate_payroll`, `apply_overrides` (بدون التصحيحات اليدوية بعد — دي في Phase 5)، وعرض النتائج.

- جدول أسعار الساعة لكل موظف.
- كروت KPI.
- جدول كشف الرواتب (`QTableView` + Model حقيقي).
- البحث والتصفية.
- الرسوم البيانية.

**✅ اختبار القبول:**
- [ ] إدخال سعر ساعة لموظف → يظهر في كشف الرواتب فورًا بنفس الرقم المحسوب في النسخة القديمة لنفس المدخلات.
- [ ] تغيير الفلاتر (بحث/حد أدنى وأقصى) → الجدول يتحدّث فورًا بدون أي تأخير ملحوظ.
- [ ] الرسوم البيانية تعكس بيانات الجدول المفلتر بدقة.

## Phase 5 — التصحيحات اليدوية وOverrides (الجزء الأكثر تعقيدًا في التفاعل)

**الهدف:** نقل شاشة تفاصيل الموظف الكاملة: Review Panel, Day Editor, Pending Changes System.

- `EmployeeDetailView` كاملة.
- نظام Pending Changes (تعديل معلّق → تطبيق/تجاهل).
- Day Editor Dialog (استبدال يوم كامل).
- Dialog تأكيد المغادرة مع تعديلات معلَّقة.

**✅ اختبار القبول:**
- [ ] تصحيح بصمة ناقصة (إدخال وقت انصراف مفقود) → يظهر في "معلَّق" أولًا، وبعد "تطبيق" ينعكس على الإجمالي فورًا.
- [ ] استبدال يوم بالكامل (Day Override) → يتجاوز البيانات الأصلية بشكل صحيح.
- [ ] محاولة الرجوع للشاشة الرئيسية مع وجود تعديلات معلَّقة → يظهر تحذير بنفس الخيارات الثلاثة (حفظ/إلغاء/تجاهل).
- [ ] مقارنة نتيجة نفس سيناريو تصحيح يدوي بين النسخة القديمة والجديدة → تطابق تام.

## Phase 7 — التصدير (Excel Export)

**الهدف:** نقل `export_to_excel` و`export_full_file` مع كل التنسيقات (ألوان، حدود، صيغ SUM).

**✅ اختبار القبول:**

- [ ] "الملف الكامل" (بالتصحيحات) يمكن إعادة استيراده والتصحيحات تُقرأ بشكل صحيح (Round-trip test).
- [ ] مقارنة بايت-لبايت (أو خلية-لخلية) بين ملف مُصدَّر من القديم والجديد لنفس المدخلات.

**الهدف:** نقل Sliders الإعدادات مع الحفظ التلقائي لكل ملف شهر.

**✅ اختبار القبول:**

- [ ] إغلاق الملف وإعادة فتحه → الإعدادات المحفوظة تُسترجع بنفس القيم.
- [ ] تغيير الإعدادات يعيد حساب الساعات فورًا بشكل صحيح.

---

## Phase 9 — التغليف النهائي واختبار الأداء على أجهزة ضعيفة

**الهدف:** exe نهائي جاهز للعميل.

- أيقونة نهائية + رقم إصدار (`APP_VERSION`).
- `PyInstaller --onefile` (أو `--onedir` لو الأول بطيء في الفتح).
- اختبار على جهاز فعلي قديم/ضعيف (أو VM بموارد محدودة: 4GB RAM، معالج قديم).

**✅ اختبار القبول:**
- [ ] زمن بدء التشغيل على الجهاز الضعيف أقل من X ثانية (يُحدَّد رقم مُرضٍ بعد القياس في Phase 0).
- [ ] استخدام الرام أثناء التشغيل العادي معقول (يُقاس ويُقارن بنسخة Streamlit).
- [ ] لا أخطاء أو Crashes عند تشغيل كل السيناريوهات السابقة على الجهاز الضعيف.

---

## Phase 10 — اختبار القبول النهائي مع العميل (UAT) والتشغيل المتوازي

**الهدف:** التأكد من الثقة الكاملة قبل التخلي عن نسخة Streamlit.

- تشغيل شهر عمل حقيقي فعلي على **النسختين معًا** (القديمة والجديدة) بنفس ملف الإدخال.
- مقارنة كشف الرواتب النهائي **رقم برقم** بين النسختين.
- جمع ملاحظات العميل على تجربة الاستخدام (فرق UX).

**✅ اختبار القبول:**
- [ ] تطابق تام لكشف الرواتب بين النسختين على الأقل لشهرين حقيقيين متتاليين.
- [ ] موافقة العميل الصريحة على الانتقال للنسخة الجديدة كنسخة أساسية.
- [ ] نسخة احتياطية من بيانات Streamlit القديمة (JSON files) محفوظة كأرشيف قبل التوقف عن استخدامها نهائيًا.
